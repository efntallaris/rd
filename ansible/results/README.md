# Aqueduct scaling experiments — shadow + main-thread merge

Eight runs across four workload variants and two client counts. All runs use
the same server build (commit on branch `aqueduct_broken`: shadow-dict
backpatch with main-thread merge), the same cluster topology (3 masters +
1 migration target), and the same three sequential migrations
(redis0→redis3, redis1→redis3, redis2→redis3 of 1365 slots each).

## What the figures show

* **`timeseries/ycsb_*.png`** — YCSB-side throughput (Kops/sec) and average
  latency over time, with the three migration bands shaded. Generated by
  `plot_ycsb_timeseries.py`.

* **`timeseries/throughput_1v2.png`** — the 1-client curve and the 2-client
  aggregate curve overlaid on a single Y-axis (300–500 Kops/sec). The
  cleanest single-figure view of "what does 2 clients buy you, and what
  happens at each migration."

* **`network/redis<N>_*.png`** — RX/TX MB/s for one host over time. The
  donor's curve dips during its own migration band; the recipient
  (redis3) climbs in three steps as it absorbs the slots from each
  donor in turn.

* **`resources/all_hosts_*.png`** — 4-host CPU/Network/Disk on shared
  axes. The Network panel here is the clearest visualization of the
  three load-redistribution steps across all donors and the recipient.

* **`resources/redis<N>_*.png`** — single-host 3-panel (CPU / Net / Disk),
  useful for diving into one node at a time.

* **`summary/throughput_comparison.png`** — the three workload variants
  side-by-side, each with both client-count curves on the same Y-axis.

## Why the numbers cluster the way they do

* **Per-node ceiling.** Each Redis instance is single-threaded for
  command execution → ~100–115 K ops/sec ceiling on this hardware
  regardless of read/write mix.

* **2-YCSB > 1-YCSB throughput.** With 50 client threads (1 YCSB) the
  cluster is *thread-bounded* and not at the per-node ceiling, so
  migrations help marginally. With 100 client threads (2 YCSB) the
  busiest node *is* at the ceiling — migrations move load off it and
  total throughput steps up.

* **All three migrations are visible per-node**, but only **M3** is
  visible in aggregate throughput at 2 YCSB: the cluster's max
  throughput is gated by the *busiest* node, and that count
  (5461 slots) only drops when M3 takes the last donor (redis2) down
  to ~4096 slots. The redistribution at M1 and M2 is real but it
  trades 5461→5461 (redis1/2 unchanged) for redis0 dropping — same
  busiest-node count, no aggregate change.

* **Workload mix barely matters here.** GET and SET both fit
  comfortably in the per-core budget at these value sizes (~100–1000 B),
  and the round-trip latency dominates. So Uniform 95/5 ≈ Uniform 50/50
  ≈ Zipfian 50/50 within ~3% in steady-state aggregate.
